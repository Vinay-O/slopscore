# Why these three paths are excluded from slopscore's own scan

slopscore holds itself to its own standard, so the exclusion list is deliberately
**minimal** — only the paths that *must* contain trigger strings by their very nature,
exactly as ESLint and Prettier exclude their own test fixtures:

- **`src/rules.js`** — the detector definitions. A linter's rule file necessarily
  contains every pattern string it looks for; scanning it would flag the patterns it is
  *defining*. (Its rule *metadata* is still the single source of truth for the catalog.)
- **`examples/`** — the deliberately-sloppy fixtures the demo runs on. `examples/slop.tsx`
  is *supposed* to score 88; that's the point of `npm run demo`.
- **`test/`** — test inputs intentionally contain trigger strings (fake secrets, injection
  payloads) so the detectors can be asserted against them.

**Everything else is scanned** — `bin/`, all of `src/` except the rule file, the config,
and any code under `docs/`. `npm run selfcheck` (`slopscore scan . --fail-on minor`) runs in
CI and must return a Slop Score of **0 — Breathtaking**, and a test (`slopscore passes its own
scan`) fails the build if slopscore ever flags itself. The tool eats its own dog food, in the
open, with nothing swept under the rug.
